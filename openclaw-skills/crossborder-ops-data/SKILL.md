# 跨境电商运营数据查询

## 触发时机

当用户询问跨境电商运营数据、经营概览、库存、低库存、缺货、利润、毛利、GMV、订单数、销量、客单价、单品销量、爆款、广告花费、利润率、运营告警或风险提示时，使用此 Skill。

当用户要求修改订单、商品、库存、价格、广告、告警状态，或要求查看原始 API payload、数据库凭据、平台 token、买家身份信息时，不使用本 Skill 执行写操作或敏感查询；应说明当前 Skill 只支持只读经营指标查询。

## 处理顺序（每次收到消息）

1. 消息去空白后整条 ∈ {`/start` `start` `开始` `操作指引` `使用指引` `指引` `help` `帮助`} → **逐字复述 onboarding 文案**（见下节），不调任何工具，结束本轮。⚠️ 这些词**不是**让你自由对话的开场——**不要**反问"你想看什么/想开始什么"，**不要**列你自己想的能力（如"定时日报""深入分析"——这些功能不存在）。看到触发词 = 复述固定 12 行文案，仅此而已。
2. 消息是切换范围/查可用范围的菜单短语（`印尼`/`全部`/`拉美`/`有哪些范围`…）→ 走「范围解析 SOP 步骤 1」：切换默认范围时**只调 `ops_set_scope_binding` 持久化**（不调任何数据工具），按那里的**固定话术**回复（如"已切换到 X"），不要把短语当话题自由发挥。
3. 否则是数据请求 → 「范围解析 SOP 步骤 2~3」定范围 → 按「意图路由」调**最少必要**的工具 → 按「回答结构」输出，首行用响应的 `scope` 字段声明范围。

约束：利润/告警**没有工具**（本期未上线），用户问到时直接如实告知，不要找或调任何工具；只调用本 skill 列出的 9 个 `ops_*` 工具，不臆造工具名或参数；**不要凭模型常识补具体数字**（如佣金率、物流/采购成本估算）——只用接口返回的数据。

## 范围解析 SOP（必须在每次调用前执行）

用户可能通过**飞书菜单按钮**（点击发送一条纯文字）或**自然语言**指定查询范围。按以下顺序判断：

### 步骤 1：判断是否为"菜单短语 = 切换默认范围"或"查询可用 scope"

如果用户消息**整条等于**以下短语（去前后空白后），则**不是数据请求**，而是切换默认范围 / 查可选范围的指令：

| 用户消息 | 含义 | 对应 scope_key | 行为 |
| --- | --- | --- | --- |
| `印尼` | 切换默认范围 | `tts-id-all` | 见下方"切换默认范围" |
| `全部` | 切换为全量 | （不传 scope_id） | 见下方"切换默认范围" |
| `拉美` | 切换（无对应 scope） | — | 触发追问 |
| `有哪些范围` / `可用范围` / `范围列表` / `list scopes` | 查询可选 scope | — | 调 `ops_scopes` 列出 |

**关键消歧**："切换范围"这类模糊短语**不进表**——它有歧义（业务范围 vs 查询维度），让用户用上面更明确的措辞。

命中时的行为：

**切换默认范围（印尼 / 全部 / 拉美）**：
- **调且仅调 `ops_set_scope_binding` 持久化默认范围**（不查 overview/inventory/orders，**也不调 ops_scopes**）：
  - 参数 `open_id` = system prompt trusted metadata 里的 `sender_id`（形如 `ou_xxx`）；`account_id` 取 metadata 的 `account_id`（没有就省略，服务端默认 `ecom-app`）。
  - `印尼` → `scope_key="tts-id-all"`；`全部` → `scope_key` 留空（切为全量）。
- 回复**必须以确认话术开头**，用工具返回的 `scope` 字段填 `{display_text}`："已切换到 **{scope}**，请问需要查什么？（GMV / 库存 / 商品 / 趋势 / 单品榜）"——`印尼` → `已切换到 **TikTok Shop / 印尼 / 1 个店铺**…`，`全部` → `已切换到 **全部范围**…`。
- ⚠️ **不要**把短语当话题反问"印尼这个方向你想看什么"或自由列维度——必须先持久化再说"已切换到 X"明确确认范围，再引导。
- 拉美：**不调任何工具**（无对应 scope，无可持久化），回复"拉美目前未配置业务范围，请告诉我具体国家（巴西/墨西哥/...）。"
- 若 `ops_set_scope_binding` 返回 400（scope 未知/停用，正常配置下不应发生）：按「异常处理」如实告知切换失败，不要谎称已切换。

**查询可用范围（有哪些范围 / 可用范围 / 范围列表 / list scopes）**：
- 调 `ops_scopes`（唯一允许调它的场景）。
- 用响应 `items[]` 渲染列表，每行 `scope_key — scope_name`，并提示"直接发送对应短语即可切换"。
- 示例：响应 `{"items":[{"scope_key":"tts-id-all","scope_name":"印尼TikTok全部店",...}],"total":1}` → 回答 `目前可用的业务范围：\n- 印尼 — 印尼TikTok全部店（tts-id-all）\n\n直接发送对应短语即可切换。`
- `scope_key` 是英文 slug，**不要让用户输入** slug；用上表里的中文短语（"印尼"等）作为用户该发的内容。

**通用约束**：菜单切换会**持久记住为默认范围**（写入 binding 表，跨会话有效）。后续不带范围词的查询自动沿用该默认（见步骤 3）；消息里带范围词则**临时覆盖本次查询、不改默认**（见步骤 2）。要改默认就再发一次范围短语 / 点菜单。

### 步骤 2：从自然语言里抽取范围词（仅当步骤 1 未命中）

> 步骤 2 抽到的范围词是**临时覆盖本次查询**，**不写 binding、不改默认**。

按下面**词表**做确定性映射，不上 LLM 抽取：

```
平台 alias：
  TikTok / TTS / 抖音小店国际 → platform=tiktok_shop
  虾皮 / Shopee → platform=shopee

国家 alias：
  印尼 / Indonesia / ID → country=ID
  马来 / Malaysia / MY → country=MY
  泰国 / Thailand / TH → country=TH
  越南 / Vietnam / VN → country=VN
  菲律宾 / Philippines / PH → country=PH
  英国 / UK / GB → country=GB
  美国 / US / USA → country=US

业务范围 alias：
  印尼 TikTok / 印尼店 / 印尼全部店 → scope_id=tts-id-all

店铺定位：
  "只看 A 店" / "只看 7494..." → 当前 scope 内收窄到该 shop_id
  "拆到店铺看" → 不改 scope，回答按店分组（聚合粒度变化，不是 scope 变化）

模糊词（必须追问，不擅自映射）：
  拉美 → 覆盖一组国家或单店，需要用户明确
```

抽取规则：
- 抽到平台/国家词 → 透传 `platform`/`country`。
- 抽到业务范围短语 → 透传 `scope_id`。
- 抽到店铺 ID 或"只看 X 店" → 透传 `shop_id`，与 `scope_id` 一起发，**服务端取交集**；越界 → 服务端 400，按「异常处理」处理。
- 抽到模糊词（拉美等）→ 追问，不调工具。

### 步骤 3：消息没带范围词 → 直接查数据（服务端自动应用默认范围）

数据请求但步骤 1/2 都没命中范围（消息里完全没提范围）时：

1. **直接调数据工具**（不调 `ops_scope_binding`，服务端已自动处理）：只要把 `open_id` = trusted metadata 的 `sender_id` 传给数据工具，服务端会自动查 binding 表注入默认范围（已切「印尼」→ 自动用 `tts-id-all`，未设过 → 全量）。
2. 回答首行用**数据工具响应**的 `scope` 字段声明范围（全量时响应 scope 为 `未限定店铺范围（全部）`），让用户知道结果覆盖什么。

## 回答首行声明范围（强制）

所有数据回答的**第一行**必须用响应的 `scope` 字段声明本次范围 + 时间窗口，例如（时间窗口取响应 `window_label`）：

```
查询范围：TikTok Shop / 印尼 / 1 个店铺 ・ 印尼时间 6/8（周一） ~ 6/9（周二），共 2 天；今天 6/9（周二）（已付款口径）
```

- `scope` 字段直接从响应取（如 `"TikTok Shop / 印尼 / 1 个店铺"`），不要自己编。
- 时间窗口：**订单/趋势/单品**（`ops_orders_summary` / `ops_orders_trend` / `ops_top_skus`）一律复述响应的 `window_label` 字段（串首带 `印尼时间`、并含星期与"今天"标注，如 `印尼时间 6/8（周一） ~ 6/9（周二），共 2 天；今天 6/9（周二）`），不要只写裸日期、更不要自己加星期或改写掉"印尼时间"前缀——**复述它即等于告知客户数据以印尼时区为准**（北京 0 点后印尼仍是前一天）；**库存/商品**类无时间窗口，用快照日期或省略。口径标签按端点定（已付款/库存快照/商品目录）。
- ⛔ **严禁自己推算或陈述"今天是周几/几号"、"某日期是星期几"**——你无法可靠知道当前日期，曾把周二误报成周一。任何星期/"今天"信息**只能来自响应的 `window_label`**；该字段没给的就不要提。"本周已 N 天/样本少"这类判断可以说（N 看 `window_label` 的"共 N 天"），但**不要附会是周几开始**。
- 同一对话后续追问，范围或窗口未变可简短"范围同上"。
- **这一行必须是回答的物理第一行**——前面不能有"好的"、空行、emoji（含 `📊`）或任何独白，直接以"查询范围"开始。**不要**用 `📊 本周 GMV…` 这类小节标题当首行顶替它。
- **全量查询（用户没提范围）也必须声明**：首行写 `查询范围：未限定店铺范围（全部）・ {时间窗口}（{口径标签}）`，不得因为"是全部范围"就省略首行。

## 意图路由（意图 → 工具）

所有数据工具的**公共参数（全可选）**：`scope_id` / `shop_ids` / `platform` / `country` / `shop_id` / `open_id`。鉴权由 openclaw 自动注入，skill 不传 token、不拼 URL。

⚠️ **`open_id` 必传**（从 system prompt trusted metadata 取 `sender_id`，形如 `ou_xxx`）：数据端点收到 `open_id` 后，**服务端自动查 binding 表取默认范围**——用户之前通过菜单切到「印尼」，后续所有数据查询**自动**限定到印尼店，agent 无需额外操作。不传则全量（除非 message 含范围词）。

**相对时间（今天/昨天/本周/上周/近7天/近30天/本月）一律传 `period` 参数，不要自己算日期**——订单类工具（`ops_orders_summary` / `ops_orders_trend` / `ops_top_skus`）支持 `period`，取值 `today` / `yesterday` / `this_week` / `last_week` / `last_7d` / `last_30d` / `this_month`，服务端按印尼时区 + 周一起算换算窗口（弱模型自己算星期容易错，故收回服务端）。只有用户给了**明确起止日期**（如"6月1日到6月7日"）时才传 `start_date` / `end_date`（`YYYY-MM-DD`，与 `period` 二选一，显式日期优先）。映射参考：`今天`→`today`、`昨天`→`yesterday`、`本周`→`this_week`、`上周`→`last_week`、`近7天/最近一周`→`last_7d`、`近30天/最近一个月`→`last_30d`、`本月`→`this_month`、`近3天`→无对应 period（传 `start_date`=今天往前2天，或仍用 `last_7d` 后截取）。

| 用户意图（关键词） | 工具 | 专有参数 / 备注 |

- "老板日报"若涉及利润/告警，必须显式告知本期不可用，不要静默缺失。

## 结果解释

- 以工具返回值为唯一事实来源；不得自行重算利润、ROI、退款率、库存覆盖等核心指标。
- **必须告知数据口径**：订单/GMV/销量类回答须复述响应的 `caliber` 字段，避免用户误以为是平台最终结算金额或内部计算值。
- 可做展示层处理（排序、筛选、截断长表、四舍五入、金额格式化）。
- 用户问接口未返回的指标 → 说明"当前接口暂未提供该指标"并给已返回的相关指标；结果为空 → 说明当前筛选下暂无数据，不臆测原因。
- 库存问题优先展示 `low_stock_items`，再展示总 SKU 数与明细。
- 趋势里连续若干天 0 是真实空窗（窗口内无单），不是数据缺失。

## 分析职责

调用工具后，把结果组织为运营分析，而非复述 JSON。回答应覆盖：

1. **事实摘要**：用返回值说明经营状态（GMV、订单数、销量、库存、低库存 SKU）。本期利润/告警无工具，不要把"毛利/告警数量"列为已知事实。
2. **异常识别**：本期无告警工具，无"系统正式告警"可引用；可基于多接口数据提"观察到的风险"（库存低但销量高、连续多日无单、低库存占比偏高等），必须标注为"基于当前数据观察"，不得称系统已生成告警。

## 异常处理

- **400**（ScopeError）：服务端拒绝了 scope/店铺组合（指定店不在 scope 内、未授权）。不要重试，如实告知范围冲突，请用户确认查哪个店/scope；可转述 `detail` 要点，但不输出 token 等敏感内容。
- **401**：内部鉴权失败（openclaw MCP token 配置问题），如实告知"数据服务鉴权异常"，不编数。
- **503**：某能力本期未上线。利润/告警本就无工具，应在路由阶段直接告知；若其他工具返 503，复述 detail，不返回 0 或编造值。
- 工具不存在 / 调用失败 / 超时：说明数据服务暂不可达（Data Hub 或 MCP 配置异常），不猜测业务数据。
- 返回字段缺失：说明接口契约与 Skill 不一致，不补造字段。
