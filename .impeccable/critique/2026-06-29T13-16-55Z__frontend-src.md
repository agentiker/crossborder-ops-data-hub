## Anti-Patterns Verdict

**LLM 评估**:核心看板(BoardPage)是真正的好活——处处 tabular-nums、真实的空/加载/错误/demo 状态、语义 token 徽章、诚实的「结算中/数据积累中」文案、无英雄大数字模板、无渐变文字、无 eyebrow、无 01/02/03。它读起来是个工具,不是 SaaS 落地页。但它不「impeccable」:它发布了自己 DESIGN.md 里点名的**头号 Absolute ban**(SectionHeader 的 `border-l-[3px]` 彩色侧条),且二级页面(Skills/Scheduled/Sidebar)退回到硬编码 `bg-amber-100 text-amber-700` 徽章。这两点正是「AI 做的,然后翻到第 3 页就不上心了」的样子。

**确定性扫描**:detect.mjs 共 20 条(16 advisory + 4 warning)。按规则:design-system-color 15、bounce-easing 2、side-tab 1、design-system-radius 1、design-system-font 1。

**交叉核对(LLM vs 检测器)**:
- **检测器漏报真违规**:它在 `index.css:196` 抓到 side-tab,但那是 `.md blockquote` 的 3px 左条——**DESIGN.md 明确豁免的排版惯例(假阳性)**。真正的违规在 `BoardPage.tsx:377` SectionHeader,检测器没覆盖 className 拼接里的 `border-l-[3px]`——靠 LLM 评审才抓到。
- **检测器抓到 LLM 略过的**:`TemplateCard.tsx:35/46` 的 `cubic-bezier(0.34,1.56,0.64,1)` 弹跳缓动(违反 ease-out 规则);`index.css:173` 的 `Sfmono-Regular` 未在 DESIGN.md 声明(代码块等宽字体,合理但应登记)。
- **多数 design-system-color 是 demo-data.ts 的假数据色**(indigo/violet/Shopify 品牌色),与 P2 demo 数据问题同源。

## Overall Impression

这是一个**值得信任的工具**,而非花架子。它的脊梁——诚实的数据状态工程(真没单 vs 加载失败、结算滞后护栏、低单量护栏)——是「数据可信即设计」在代码里的真实兑现,不是口号。最大的机会不在加东西,而在**收敛一致性**:让二级页面的徽章/卡片/阴影回到看板已经做对的那套语义 token 系统,并清除自己 DESIGN.md 点名禁止的侧边色条。把这些自伤项修掉,28 分能直接进 32+。

## What's Working

1. **诚实的数据状态工程是产品脊梁,且做得极好。** `NoDataBanner`(BoardPage.tsx:277)区分「真没单」与「加载失败」;`settlingNote`/`adComplete` 护栏(1098–1100);`coverageIncomplete`/`costMissing` 折叠进低强度图标行而非堆一面 amber 墙。这正是「可信度本身是核心体验」的代码兑现。
2. **InfoTooltip portal 方案是品类级的。** portal→body + `position:fixed` + 视口翻转/夹紧 + capture 阶段滚动重定位(ui/tooltip.tsx),正确解决了滚动容器看板里的结构性裁剪问题。「短信息用 tooltip、密集信息用点击弹窗(AdSpendDialog)」的升级规则是对的产品直觉,不是 modal-first。
3. **色盲安全语义是真的实现了,不是嘴上说。** `BurstBadge` 配 Flame 图标+文字+色(1928);MetricCard delta 用 ↑/↓/− 字形+色(1073);渠道饼图图例带名+%。在多数仪表盘偷懒的地方它守住了 Quiet Semantics Rule。

## Priority Issues

**[P0] SectionHeader 彩色侧边条违反项目自己点名的头号 Absolute ban。**
- **What**:`SectionHeader`(BoardPage.tsx:374–378)渲染 `border-l-[3px] border-info bg-info/5`(区2 强调)和 `border-l-[3px] border-border-shallow`(区1)。DESIGN.md 第 247 行明确禁止 >1px 的彩色 `border-left/right` 强调,仅豁免 markdown 引用块。impeccable Absolute bans 把它列在第一条。
- **Why it matters**:这正是整个品牌定位要对抗的教科书级 AI-slop 标记。它还承载着区1/区2 的区分职责,所以不是无意装饰——一个被刻意写进最常看组件里的禁令违规,流利用户一眼就能看出。
- **Fix**:把 3px 左条换成被许可的替代:区2 用整块 `bg-info/5` 浅底 + 前导 `Info` 图标 + 标签;区1 仅留标题或一道 `border-b` 发丝线。浅底+图标已承载「不随筛选/监控」语义,侧条是冗余。
- **Suggested command**:`$impeccable quieter`

**[P1] 硬编码 Tailwind 调色板徽章违反 Semantic Badge Rule,散落 5+ 文件。**
- **What**:`bg-amber-100 text-amber-700`(Sidebar.tsx:127 待开发、ScheduledPage.tsx:73、SkillsPage.tsx:44)、`bg-green-100 text-green-700`(TaskCard.tsx:41 运行中)、`bg-orange-100 text-orange-700`(SkillCard.tsx:67、SkillDetailDialog.tsx:55),外加 `text-blue-500/green-500/orange-500` 图标(ChatMessage.tsx:33–36、WelcomeScreen.tsx:42)。DESIGN.md Semantic Badge Rule:「绝不用 Tailwind 现成调色板…硬编码即破坏色彩统一。」
- **Why it matters**:这些正是让 app 不再像一套设计系统的颜色。看板用 `bg-caution/15 text-caution`、`bg-positive/15 text-positive` 是对的——于是同一个「状态 pill」概念在不同页面渲染成两种视觉。这是一致性启发式在用户眼里失败。
- **Fix**:映射到 token——待开发→`bg-caution/15 text-caution`;运行中→`bg-positive/15 text-positive`;已暂停→`bg-fill text-foreground-secondary`;图标色→`text-info`/`text-positive`/`text-warning`。一次清扫。
- **Suggested command**:`$impeccable colorize`

**[P2] 6 格 KPI 网格 + 死掉的 ROI/ROAS 格稀释了「一眼可读」的顶层。**
- **What**:`BusinessOverview` KPI 网格(BoardPage.tsx:1240–1266)显示 6 格;「ROI」硬编码 `value="—" subtitle="口径待定"`,ROAS 常显「—/结算中」。6 个英雄位里 2 个长期或经常为空。
- **Why it matters**:设计原则 1 说关键信号优先、次要让位。一个永远写着「口径待定」的格子是反信号——训练眼睛跳过整个网格。手机端(2 列)它是老板看到的第一屏。
- **Fix**:ROI 在有口径前直接撤掉(别在英雄行发占位格);网格收为 4 个真指标(GMV/订单/销量/广告消耗)+ ROAS 仅在 `adComplete` 时作第 5 个。把死指标降出英雄行。
- **Suggested command**:`$impeccable distill`

**[P2] 「可信」仪表盘里的 demo 数据 tab 动摇核心品牌承诺。**
- **What**:`OrderSection`(下单趋势/退货/退款)与 `BusinessOverview`(流量/漏斗)发布捏造的 Shopify/Amazon/TikTok 序列,藏在「演示数据」pill 后(BoardPage.tsx:2499–2544、demo-data 导入)。
- **Why it matters**:整个产品是「数据可信即设计」。老板点进关于竞品平台(Shopify/Amazon——这家店只在 TikTok ID 站)的假数据,学到的是仪表盘混着真假。演示徽章诚实,但认知代价(4-tab 里一半通向假数据)超过占位价值。
- **Fix**:要么移除 demo tab 改成单张「退货分析·数据源开发中」空状态教学卡,要么用 dev flag 门控。别让假数据和待发货(真实且运营关键)做平级 tab。
- **Suggested command**:`$impeccable distill`

**[P3] `shadow-xl`/`bg-white` 弹层违反 Soft-Lift 与 No-Ghost-Card 规则。**
- **What**:AskAiSheet(`bg-card shadow-xl`,line 203)、AppShell 移动侧栏(`shadow-xl`,line 110)、skills/scheduled 的 `bg-white shadow-lg` 弹窗(SkillDetailDialog.tsx:20、SelectTemplateDialog.tsx:15、各 dropdown)。DESIGN.md 阴影上限 `shadow-sm`(blur≤2px),且 `bg-white`(非 `bg-card`)绕过暖色卡片 token,显得比看板更冷。
- **Why it matters**:严重度较低(弹层确实需要浮起),但 `bg-white` + `shadow-lg` + `border` 同元素正是 ghost-card 套路,且冷白破坏橄榄暖白基调。
- **Fix**:统一一个 `Dialog`/`Sheet` 表面 token,`bg-card` 上至多 `shadow-lg`;永不 `bg-white`+`shadow-lg`+`border` 三连。
- **Suggested command**:`$impeccable polish`

## Persona Red Flags

**Alex(高级运营,PC,重度盯盘)**:
- 全程无键盘快捷键/命令面板——每个筛选/tab/排序都得用鼠标。对整天处理告警的人是 Linear 级工具已消除的摩擦。
- 库存明细表是真正的工作台,却无行点击操作、无批量选择、无导出。能排序,但对缺货 SKU 做不了任何事(无补货链接)。
- 4-tab 里 2–3 个是 demo:Alex 学会无视它们,白占屏幕。

**Sam(无障碍)**:
- 对比度风险:`text-foreground-tertiary`(`rgba(25,36,32,0.5)`)被用于真实内容而非纯装饰——TaskCard.tsx:49/54 把排程标签和描述放进 tertiary(≈0.5 alpha 压在橄榄白上,临界或低于 4.5:1),违反自己的 Readable Ink Rule。
- 自定义 DateRangePicker 日期网格可 Tab 到达,但无方向键导航(要 Tab 过 30 个按钮)。
- iOS 开关(SkillCard/TaskCard)是 `role="switch"`+`aria-checked`(好),但旋钮 `bg-white shadow-md` 在开关本体上无 focus-visible 环。

**老板(碎片时间,手机看大盘)**:
- 筛选栏滚动收起(省空间是好的),但活动店铺/范围 chip(`范围 · {scope}`)是 `hidden sm:block`(BoardPage.tsx:200)——手机上老板不滚回顶部就看不到这是哪家店的数。对多店老板,「这是谁的 GMV?」是真实歧义。
- 手机第一屏是 6 格 KPI 网格,ROI/ROAS 读作「—」——碎片时间一瞥落在空格上。

**一线运营(告警处置)**:
- 看板把缺货/超时/费率异常呈现得很美,但**从告警到处置没有任何动作入口**。库存明细表显示可售天数和状态徽章,却不链接补货流。「告警为被处置而生」(原则 3)只建了一半:信号出色,下一步动作在 UI 里缺席。

## Questions to Consider

1. **为什么 demo 数据 tab 要和真实运营数据做平级 tab?** 如果答案是「为了在客户面前显得完整」,那正是这个品牌「数据可信即设计」原则要拒绝的交易——眼前的感知完整 vs 永久侵蚀的数据信任。如果流量/漏斗/退货/退款变成诚实的「数据源开发中」卡,会坏掉什么?
2. **手机上老板看的是谁的数?** 范围 chip 是桌面端独占,筛选栏又滚走。对碎片时间的多店老板,店铺身份会不会有歧义——粘性手机顶栏是否该带上活动范围,而不只是「数据中枢」?
3. **告警通向哪里?** 仪表盘极擅长「展示」缺货/超时/费率异常,却没有任何 UI 内通向「动作」(补货/改发货)的路径。如果原则 3「告警为被处置而生」是真的,补货链接从库存表缺席的刻意理由是什么——对坐在 PC 前正看这屏的运营,「它去飞书」是可接受的答案吗?
